01/The challenge

Manual freight audit is slow and error-prone

Every row needs a person to read a dozen columns, decide what's off, and type the right reason in the Anmerkung column. Different carriers, different rules, every single week.

Time
Hours per file
Hundreds of rows × multiple carriers × multiple files per week.
Consistency
Wording drifts
Two auditors write the same finding in different words. Reports become hard to aggregate.
Accuracy
Tiny deltas hide
A €0.40 difference is easy to overlook on row 423 of sheet 7.

Missed

Overfired

Export the result as CSV or JSONL for analysis or downstream training.

04/Architecture

End-to-end pipeline

Five stages from the file you drop to the file you download — all running inside one browser tab.

01 · INPUT
Read bytes
The browser's FileReader turns the .xlsx into an ArrayBuffer in memory.
Built-in · File API
02 · PARSE
Workbook → rows
Each sheet becomes a list of row objects keyed by column header (Frachtdiff, SNK Kosten DL, …).
SheetJS · xlsx 0.18.5
03 · DECIDE
Rule engine
A per-carrier dispatcher walks each row through an ordered ladder of rules. First match wins.
Custom · anmerkung.js
04 · PATCH
Surgical XLSX edit
Treat the .xlsx as a ZIP, rewrite only the Anmerkung cells in sheetN.xml, re-zip.
JSZip · 3.10.1
05 · OUTPUT
Blob → download
The patched archive becomes a Blob; the browser saves it with the original filename.
Built-in · Blob API

Stack: vanilla HTML / CSS / JS · zero build step · two CDN libraries · service worker for offline.

04/Architecture

Rule engine design

Each carrier has its own function — processDachser(), processKN(), processDHL(), processWackler(). Inside each is an ordered ladder of rules; the first match wins, the rest are skipped.

1
Specific before generic Narrower conditions are evaluated first so they aren't masked by broader fallbacks.
2
Single phrase catalog Every output string lives in one PHRASES object. Changing a wording is a one-line edit, applied everywhere.
3
Tunable tolerance Differences below a per-carrier threshold (e.g. €0.08) count as zero. Stored in localStorage.
assets/anmerkung.js · processWackler() · simplified
function processWackler(row) {
  // 1. duplicate Fremdnummer detected
  if (isDuplicateFremdnummer(row))
    return "Fremdnummer Doppelt berechnet";

  // 2. B2C-Line would have been cheaper
  if (b2cWouldBeCheaper(row))
    return "hätte B2C-Line abrechnen dürfen";

  // 3. SNK threshold buckets (38 / -11.5 / 22)
  if (snkDiff(row) > T_WACKLER)
    return "SNK Differenz";

  // 4. weight or volume mismatch
  if (weightOff(row))
    return "Differenz aufgrund abweichender Gewichte";

  // 5. fallback — leave empty
  return "";
}

Production ladders are longer: Dachser ≈ 22 rules, K+N ≈ 18, DHL ≈ 15, Wackler ≈ 14.

04/Architecture

Why we patch the file in place

Letting a library re-write the entire workbook would silently lose styles, formulas, and embedded drawings. So we don't.

★
An .xlsx is a ZIP archive Rename it to .zip and you can browse the XML inside. We use that fact directly.
★
Only the Anmerkung column changes JSZip rewrites a few <c> cells in sheetN.xml and leaves every other byte alone.
★
Styles, formulas, pictures survive Conditional formatting, charts, merged cells, frozen panes — all preserved exactly.
★
Diff-able output Open the original and the processed file side-by-side: only the one column differs. No surprises.

05/Trust & deployment

Your data stays in your browser

There is no backend. There is no account. There is nothing to log in to.

Privacy
100% client-side
The .xlsx is read, processed, and patched entirely in your browser tab. There is no backend to send it to.
Availability
Works offline
After your first visit, the page and its libraries are cached. Open it without internet and it still runs.
Deployment
Static hosting
A single HTML page on GitHub Pages. No accounts, no logins, no cookies, no infrastructure to operate.
Auditability
Source is open
Every rule lives in assets/anmerkung.js. Nothing is hidden behind an API.
